Origin Context

CRM Wiki → System Evolution

This system originated in 2023 as an attempt to build a wiki-like extension on top of an existing CRM to support a very young sales team with better internal context and structure.

The original intent was knowledge accessibility, not automation.

As implementation progressed, operational constraints exposed deeper system-level opportunities:

  • CRM data could trigger workflows instead of only storing records
  • Onboarding could become event-driven rather than manual
  • Content generation could be partially system-assisted

This shifted the project from internal tooling into a production exploration of AI-augmented operational systems.

System Overview

Building a Modular AI-Native Operations System in a Small Agency Environment

Overview

A small digital marketing agency was facing a familiar scaling problem: increasing client demand, slow onboarding cycles, inconsistent content production, and limited operational capacity without proportional increases in headcount.

To address this, I designed and implemented an internal system that combined existing CRM infrastructure with API-based AI tools to automate and standardize core parts of client onboarding, content generation, and internal sales workflows.

The system was built iteratively inside a real production environment, primarily by a single developer, using modular architecture principles and existing SaaS components rather than custom infrastructure.

The goal was not full automation. The goal was to reduce operational friction and increase throughput while maintaining human oversight in all client-facing outputs.

Context & Constraints

  • No dedicated engineering team at project start
  • Limited budget (API + SaaS tooling)
  • Evolving requirements from non-technical stakeholders
  • Single primary builder
  • Human approval required for all outputs

System Architecture Overview

The system was built as a modular API-driven workflow layer integrated into an existing CMS/CRM environment.

  • Modular service decomposition instead of monolithic architecture
  • Event-driven triggers for onboarding and content generation
  • Batch optimization cycles
  • Human-in-the-loop validation gates
  • Interchangeable AI tooling abstraction layer

Core principle: replace rigid workflows with composable, event-driven systems connected through APIs.

Event-Driven Client Onboarding

  • CRM record creation

  • Onboarding form generation
  • Website/content scraping
  • AI-assisted brand profile generation

  • Human review gate

Operational Outcomes

  • Onboarding reduced from ~60 days → ~3 weeks
  • Support response time under 2 hours across ~90 clients
  • ~4x revenue growth within ~70 days after scaling deployment
  • Reduced operational load without proportional headcount increase

Constraints, Failures, and Limitations

  • API cost scaling constraints
  • Noisy scraped data inputs
  • Dependency on external model providers
  • Increasing orchestration complexity over time
  • Human validation required for reliability

The system was intentionally not fully autonomous. It functioned as a human-stabilized automation layer rather than a replacement for operators.
